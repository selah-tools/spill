# Runbooks

## Dashboards and alerts

- **GitHub Actions:** https://github.com/selah-tools/spill/actions
- **Vercel deploys:** https://vercel.com/selah-tools/spill
- **Sentry errors:** configure `SENTRY_ORG` and `SENTRY_PROJECT` in GitHub variables, then access at https://sentry.io
- **Alerting:** Failed Quality, Integration, Release Plan, and Vercel Deploy runs on `main` automatically open or update a GitHub Issue with a response checklist via `.github/workflows/workflow-alerts.yml`. Issues are labeled `ci`.
- **Vercel deploy status:** each deploy is visible at the Vercel dashboard. Preview deploys run on every PR; production deploys run on every push to `main`.

## Checking deploy impact

After a production deploy lands on `main`:

1. Open the Vercel deploy dashboard and confirm the deploy succeeded.
2. Check Sentry for new errors or spikes in the `production` environment since the deploy timestamp.
3. Review the GitHub Actions Quality run that accompanies the deploy — confirm all checks pass.
4. If errors appear in Sentry, use the "Suspect Commit" feature to trace them to a specific PR.
5. If the deploy itself failed, check the auto-created workflow alert issue for the run link.

## Feedback writes are failing

1. Check the Vercel function logs for `/api/feedback`.
2. Grab the `X-Request-Id` from the failing response.
3. Confirm `KV_REST_API_URL` and `KV_REST_API_TOKEN` are set in the target Vercel environment.
4. Re-run the request and search logs by `requestId`.
5. Verify Upstash write health with the `check-kv` agent skill in `.agents/skills/check-kv/SKILL.md`.
6. Use the request ID to correlate the failing call across Vercel logs and Upstash REST traces.

## Question map looks stale

1. Confirm `app/questions.json` changed in the deploy you expect.
2. Run `pnpm test` to verify question-map tests still pass.
3. Deploy or redeploy through the Vercel workflow.
4. Hit `/api/question-map` directly and inspect the `count` field.

## Resetting the feedback KV

If you need to clear Spill's Upstash KV state entirely (for example, to reset feedback counters during testing):

1. Run `python3 scripts/reset-kv.py --dry-run` to inspect the keys that would be deleted.
2. Run `python3 scripts/reset-kv.py` to delete the full keyspace.
3. If you want production assets refreshed immediately afterward, trigger a production deploy.

## Quality gate is red

1. Run `pnpm quality` locally.
2. Fix the first failing step before rerunning everything.
3. If `pnpm size:check` fails, either shrink the file or explicitly update the budget in `scripts/check-file-sizes.mjs` with justification in the PR.
4. If `pnpm tech-debt` fails, either remove the marker or link the follow-up in code review and update the scanner policy.

## Preparing a release

1. Add a changeset: `pnpm changeset`
2. Merge to `main`
3. Review the release PR generated by GitHub Actions
4. Verify the production Vercel deploy
5. If the deploy fails, check the auto-created workflow alert issue for the failing run link

## Sentry integration

- **Frontend:** `app/sentry.ts` initializes `@sentry/browser` when `VITE_SENTRY_DSN` is set. Captures unhandled errors with breadcrumbs and replay on error.
- **API:** Edge handlers currently rely on structured logs plus `X-Request-Id` correlation in Vercel and Upstash. They are not wired to send exceptions to Sentry yet.
- **Source maps:** The `vercel.yml` workflow notifies Sentry of new releases via `getsentry/action-release@v1`, enabling "Suspect Commit" linking in browser error events.
- **Required secrets:** `SENTRY_AUTH_TOKEN` (Actions secret), `SENTRY_ORG` and `SENTRY_PROJECT` (Actions variables), `VITE_SENTRY_DSN` (Vercel env var).
